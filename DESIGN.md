# 설계 결정 문서 — Eth Paduck Wallet

이 문서는 **"왜 이렇게 만들었는가"** 를 설명한다.
각 결정에 대해 어떤 문제가 있고, 왜 문제인지, 어떻게 막을 것인지, 왜 이 방법을 선택했는지를 기록한다.

코드 안의 주석과 함께 읽으면 전체 설계 의도를 파악할 수 있다.

---

## 1. 전체 아키텍처: 백엔드 중계 구조 (구조 B)

### 어떤 문제인가

클라이언트(React)가 Alchemy RPC에 직접 접속하는 구조(구조 A)에서는:

```
브라우저 → Alchemy (https://eth-sepolia.g.alchemy.com/v2/API키)
```

브라우저 개발자 도구 Network 탭을 열면 이 URL이 그대로 보인다.
API 키를 추출하는 데 걸리는 시간: **30초 이내**.

추출된 API 키로 공격자는:
- 자기 프로젝트에서 우리 API 키를 무제한 사용
- 스크립트로 대량 호출해 우리 Alchemy 한도를 소진
- 유료 플랜이면 예상치 못한 과금 발생

이는 코드 난독화로 막을 수 없다. 요청이 나가는 순간 실제 URL이 노출되기 때문이다.

### 왜 이게 문제인가

Alchemy API는 회사의 인프라 자원이다.
노출된 키로 외부인이 이 자원을 무제한 사용하면:
- 서비스 중단 (한도 소진)
- 예상치 못한 과금
- 우리 서비스의 정상 사용자가 피해를 입음

### 어떻게 막는가

**백엔드 중계 구조(구조 B)** 를 채택한다:

```
브라우저 → 우리 Node.js 서버 → Alchemy
```

- 클라이언트는 `/api/balance?address=0x...` 만 호출한다
- Alchemy URL과 API 키는 서버 환경변수에만 존재한다
- 클라이언트는 Alchemy가 어디 있는지조차 알 수 없다

### 왜 이 선택인가

구조 B는 API 키 노출 문제를 **근본적으로** 해결한다.
다른 보호 패턴(rate limiting, 캐싱)도 이 구조 위에서만 효과가 있다.
구조 B 없이는 나머지 보호가 의미없다.

## 2. 입력값 검증 (시나리오 3: 잘못된 입력값)

### 어떤 문제인가

검증 없이 모든 입력을 Alchemy에 전달하면:

```
GET /api/balance?address=<script>alert(1)</script>
GET /api/balance?address=aaaaaaaaaa
GET /api/balance?address=
GET /api/balance?address=SELECT * FROM users
```

각 요청마다 Alchemy를 호출한다. Alchemy는 "invalid address" 에러를 반환하지만, **API 요청 횟수는 카운트된다**.
매번 다른 잘못된 값을 보내면 캐시 히트가 없어 모든 요청이 Alchemy에 도달한다.

또한 `<script>` 태그가 응답에 포함되어 HTML에 렌더링되면 XSS 공격이 된다.

### 왜 이게 문제인가

API 요청 한도가 낭비된다. 잘못된 입력에 대해 우리가 비용을 지불하게 된다.

### 어떻게 막는가

Alchemy를 호출하기 **전에** 주소 형식을 검증한다.
검증 실패 시 즉시 400 Bad Request를 반환하고, Alchemy 호출은 일어나지 않는다.

### 왜 ethers.isAddress()인가

선택지 A: 정규식 `/^0x[0-9a-fA-F]{40}$/`
- 장점: 외부 의존성 없음, 단순
- 단점: 형식만 체크. 체크섬 오류를 감지하지 못함

선택지 B: `ethers.isAddress()`
- 장점: 체크섬(EIP-55)까지 검증. ethers.js는 Alchemy 호출에 이미 필요하므로 추가 의존성 없음
- 단점: 없음

ethers.js는 어차피 `blockchain.js`에서 사용하므로 `ethers.isAddress()`를 쓰는 것이 의존성을 전혀 추가하지 않으면서 더 엄격한 검증을 제공한다.

### 추가: 주소 정규화 (시나리오 5: 캐시 오염)

`ethers.getAddress(address)` 로 EIP-55 체크섬 형식으로 정규화한 뒤 캐시 키로 사용한다.

대소문자만 다른 세 주소는 모두 동일한 계정이다:
```
0xd8da6bf26964af9d7eed9e03e53415d37aa96045  (소문자)
0xD8DA6BF26964AF9D7EED9E03E53415D37AA96045  (대문자)
0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045  (체크섬)
```

정규화 없이 캐시 키로 쓰면 3개의 분리된 캐시 엔트리가 생겨 Alchemy를 3번 호출한다.
정규화하면 항상 같은 키 → 캐시 히트율 극대화.

## 3. Rate Limiting (시나리오 2: 무차별 반복 호출)

### 어떤 문제인가

공격자는 우리 API 엔드포인트를 알게 된 순간 스크립트로 대량 호출할 수 있다:

```python
# 브라우저 Network 탭에서 "Copy as cURL" 한 번이면 이 스크립트가 만들어진다
while True:
    requests.get("http://우리서버/api/balance?address=0x0000...")
```

스레드 50개로 실행하면 초당 수천 건의 요청이 서버에 도달한다.

### 왜 이게 문제인가

- 각 요청이 Alchemy 호출로 이어지면 API 한도를 수초 만에 소진한다
- 서버 CPU/메모리 과부하로 정상 사용자도 서비스를 이용할 수 없게 된다

### 어떻게 막는가

IP 기반으로 단위 시간당 요청 수를 제한한다.
설정값: **1분에 10회**
초과 시: 429 Too Many Requests + `Retry-After` 헤더

### 왜 1분에 10회인가

정상 사용자의 행동 패턴:
- 지갑 잔액을 수동으로 확인할 때 1분에 1~2번이 자연스럽다
- 10번은 정상 사용에 여유를 두면서도 자동화 공격(수백~수천 회)과 명확히 구분된다

### 왜 라이브러리(express-rate-limit)를 쓰지 않았는가

- 의존성 최소화: 꼭 필요한 패키지만 사용한다는 원칙
- 동작 원리를 완전히 이해하고 직접 제어한다
- 이 과제 규모에서 직접 구현으로 충분하다

### 왜 IP 기반인가 (한계 인식)

로그인 시스템이 없으므로 사용자를 식별할 방법이 IP뿐이다.

**알고 있는 한계:**
1. **VPN/프록시 우회**: 결심한 공격자는 IP를 바꿔가며 우회 가능하다
2. **NAT 환경**: 학교·회사에서 여러 사람이 같은 외부 IP를 공유한다.
   한 명이 한도를 채우면 같은 IP의 다른 정상 사용자도 차단된다.
3. **IPv6**: 주소 공간이 너무 넓어 IP 기반 제한이 덜 효과적이다.

이 한계에도 불구하고 IP 제한은 **대부분의 무차별 공격을 막는 데 충분히 효과적**이다.
더 정밀한 제어가 필요하면 사용자 인증 후 토큰 기반 rate limiting이 필요하지만,
이는 이 과제의 범위를 벗어난다.

### 왜 고정 윈도우(Fixed Window)인가

슬라이딩 윈도우는 "직전 N분" 동안의 요청을 정밀하게 추적하지만 구현이 복잡하다.
고정 윈도우는 단순하고, 이 과제 규모에서 발생하는 경계 공격(0:55~1:05에 20번)은
현실적으로 큰 위협이 아니다.

## 4. 캐싱 (시나리오 4: 동일 주소 반복 조회)

### 어떤 문제인가

악의 없는 사용자도 Alchemy를 낭비시킬 수 있다:
- 새로고침 반복
- 우리 API를 자신의 서비스에 붙여서 주기적 호출
- 1초마다 잔액을 모니터링

Ethereum Sepolia 블록 생성 주기: **약 12초**
→ 잔액은 최소 12초에 한 번씩만 바뀔 수 있다
→ 1초마다 조회하면 12번 중 11번은 의미없는 Alchemy 호출이다

### 어떻게 막는가

동일한 주소에 대한 반복 요청은 캐시에서 반환하고 Alchemy를 호출하지 않는다.

### TTL: 30초

| TTL | 장점 | 단점 |
|-----|------|------|
| 5초 | 거의 실시간 | Alchemy 호출 많음 |
| **30초** | **호출 30배 절약. 블록 약 2.5개 분량** | 최대 30초 지연 가능 |
| 60초 | 60배 절약 | 잔액 변화가 1분 뒤 반영 |

잔액 조회 서비스의 특성상 30초의 지연은 허용 가능하다.
잔액이 1~2블록 뒤처져도 사용자 경험에 큰 영향이 없다.

### 왜 인메모리 캐시인가 (Redis 대신)

| 방식 | 장점 | 단점 |
|------|------|------|
| 인메모리(Map) | 추가 의존성 없음, 매우 빠름, 설정 불필요 | 서버 재시작 시 초기화, 단일 서버만 가능 |
| Redis | TTL 내장, 재시작 후 유지, 다중 서버 공유 | 별도 서버 설치·운영 필요, 이 규모에 과함 |

이 과제는 단일 서버이므로 인메모리로 충분하다.
서버 재시작 시 캐시가 초기화되지만, 이는 서비스 중단이 아닌 일시적 성능 저하일 뿐이다.

### 캐시 크기 제한

무한정 메모리 증가를 막기 위해 최대 1,000개 엔트리로 제한한다.
초과 시 가장 오래된 항목부터 삭제한다(LRU 근사).

## 5. 에러 처리 (시나리오 6: 정보 노출)

### 어떤 문제인가

에러를 그대로 반환하면 내부 구현이 노출된다:

```javascript
// 나쁜 예
res.status(500).json({ error: error.message });
// → "could not detect network (event="noNetwork", version=6.9.0)"
//   라이브러리 이름과 버전이 노출된다
```

공격자는 이 정보로:
- 사용 중인 라이브러리의 알려진 취약점을 노린다
- 내부 네트워크 구조를 파악한다

### 어떻게 막는가

```
서버 로그: 상세 에러 전체 기록 (운영자용)
클라이언트 응답: 일반적인 한국어 메시지만 반환
```

### HTTP 상태 코드를 정확하게 쓰는 이유

| 상황 | 상태 코드 |
|------|---------|
| 주소 형식 오류 | 400 Bad Request (클라이언트 잘못) |
| Rate limit 초과 | 429 Too Many Requests |
| Alchemy 실패 | 503 Service Unavailable (외부 서비스 장애) |
| 예상치 못한 서버 오류 | 500 Internal Server Error |

400인데 500을 반환하면 클라이언트가 "서버 버그"로 오해한다.
503은 "잠시 후 재시도"가 의미 있다는 신호이므로, Alchemy 장애에는 503이 정확하다.

## 6. 미들웨어 순서의 중요성

요청 처리 흐름:
```
① validateAddress → ② rateLimitMiddleware → ③ 캐시 확인 → ④ Alchemy 호출
```

**① → ② 순서가 중요한 이유:**

만약 rate limit이 먼저 온다면, 공격자가 잘못된 주소를 계속 보내서 rate limit 카운트를 소모시키고 결국 정상 사용자의 유효한 요청을 차단하는 공격이 가능해진다.

형식이 잘못된 요청은 rate limit 카운트에 포함시키지 않는다.

## 7. CORS 설정

프론트엔드(Vite, `localhost:5173`)와 백엔드(`localhost:5000`)는 다른 출처(Origin)다.
브라우저는 보안상 다른 출처의 API를 기본으로 차단한다(Same-Origin Policy).

**개발 환경**: Vite proxy(`vite.config.js`)가 `/api` 요청을 백엔드로 전달하여 CORS 문제를 우회한다.
**백엔드**: cors 미들웨어로 `ALLOWED_ORIGIN`만 명시적으로 허용한다.
`origin: "*"` (와일드카드)는 어떤 출처에서든 이 API를 호출할 수 있어 운영에서 위험하다.

## 8. 한계 및 개선 가능한 점

이 서비스가 가진 한계를 솔직하게 기록한다.
한계를 인식하고 선택했다는 것이 이 문서의 핵심이다.

| 한계 | 원인 | 개선 방법 |
|------|------|---------|
| IP 기반 rate limit → VPN 우회 가능 | 사용자 인증 없음 | 토큰 기반 인증 + 토큰별 rate limit |
| 인메모리 캐시 → 서버 재시작 시 초기화 | Redis 미사용 | Redis 도입 |
| 고정 윈도우 rate limit → 경계 취약점 | 단순 구현 우선 | 슬라이딩 윈도우 구현 |
| NAT 환경에서 정상 사용자 차단 가능 | IP 기반 한계 | 사용자 인증 도입 |
| 단일 서버 → 스케일아웃 불가 | 인메모리 상태 | Redis + 로드밸런서 |

## 9. 비용 발생 지점 요약

```
요청 수신
    │
    ▼
① 입력값 검증 ────────────── 실패 → 400 반환 (비용: 없음)
    │ 통과
    ▼
② Rate Limit 확인 ─────────── 초과 → 429 반환 (비용: 없음)
    │ 통과
    ▼
③ 캐시 확인 ────────────────── 히트 → 200 반환 (비용: 없음)
    │ 미스
    ▼
④ Alchemy 호출 ──────────────────────────────── ★ 비용 발생
    │
    ▼
캐시 저장 → 200 반환
```

**비용이 발생하는 지점은 딱 하나: Alchemy를 호출하는 순간.**
①②③은 모두 이 지점 앞에 세우는 방어선이다.
