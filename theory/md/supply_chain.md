# 공급망 공격 (Supply Chain Attack) — npm 패키지 의존성

---

## 1. 공급망이란 무엇인가

"공급망(Supply Chain)"이라는 단어는 원래 제조업에서 왔다.

자동차를 만들려면 철, 유리, 고무, 반도체 등 수백 가지 부품이 필요하다.
이 부품들은 각각 다른 공급업체에서 온다.
자동차 회사가 아무리 품질 관리를 잘 해도,
**부품을 납품하는 공급업체에 문제가 생기면 최종 제품도 망가진다.**

소프트웨어도 똑같다.

내 코드를 아무리 안전하게 짜도,
내가 `npm install`로 가져다 쓰는 패키지에 악성 코드가 심어져 있으면
내 서비스 전체가 뚫린다.

이게 **소프트웨어 공급망 공격**이다.

---

## 2. npm 생태계는 왜 특히 위험한가

### 의존성의 의존성

`npm install express` 한 줄을 실행하면 무슨 일이 일어나는지 확인해보자.

```bash
npm install express
added 64 packages
```

express 하나를 설치했는데 64개가 설치된다.
내가 직접 선택한 건 express 1개지만, express가 의존하는 패키지들이 줄줄이 딸려온다.

조금 더 큰 프로젝트라면:
```bash
npm install react react-dom express ethers
added 312 packages
```

내가 선택한 4개 → 실제로 설치된 건 312개.

이 312개 중 하나라도 악성 코드가 들어있으면?

### 신뢰의 연쇄

```
내 코드
 └── express (내가 신뢰한다)
      └── body-parser (express가 신뢰한다)
           └── bytes (body-parser가 신뢰한다)
                └── ... (내가 이름도 모르는 패키지들)
```

나는 express를 신뢰하지만, express 개발자가 신뢰하는 모든 패키지를 내가 검증할 수는 없다.

### npm 생태계의 규모

2024년 기준:
- npm 레지스트리에 등록된 패키지: 250만 개 이상
- 주간 다운로드 수: 수백억 건
- 패키지 유지 관리자 1명인 패키지: 전체의 상당수

개인 개발자 한 명이 수백만 개발자들이 쓰는 패키지를 혼자 관리하는 경우가 많다.
이 유지 관리자의 계정이 탈취되거나, 유지 관리자가 나쁜 의도를 가지면?

## 3. 실제 사례들

### event-stream 사건 (2018)

`event-stream`은 Node.js 스트림 유틸 패키지로, 주간 200만 건 이상 다운로드되던 패키지였다.

원래 유지 관리자가 더 이상 관리하기 싫다고 올린 글에,
누군가가 "내가 관리하겠다"고 나섰고, 유지 관리자가 권한을 넘겼다.

새 유지 관리자는:
1. `flatmap-stream`이라는 새 패키지를 만들어 악성 코드를 숨겼다
2. `event-stream`의 의존성에 `flatmap-stream`을 추가했다
3. 악성 코드는 특정 암호화폐 지갑 앱(Copay)을 탐지해서 개인키와 잔액을 탈취하도록 설계됐다

수백만 개발자가 모르는 사이에 악성 코드가 포함된 버전을 설치했다.
발견까지 수주일이 걸렸다.

**이 공격의 타겟이 바로 암호화폐 지갑이었다.**

### colors & faker 사건 (2022)

두 패키지의 개발자 Marak Squires가 오픈소스 기업 착취에 반발하여
자신의 패키지에 고의로 무한루프와 무의미한 텍스트를 출력하는 코드를 배포했다.

- `colors`: 주간 2000만 건 이상 다운로드
- `faker`: 주간 300만 건 이상 다운로드

이 패키지들에 의존하던 수천 개의 프로젝트가 갑자기 망가졌다.
악성 코드가 아닌 고의적 파괴였지만, 피해는 동일했다.

### node-ipc 사건 (2022)

러시아의 우크라이나 침공 직후, `node-ipc` 패키지 유지 관리자가
러시아/벨라루스 IP에서 접속하는 경우 파일을 삭제하는 코드를 배포했다.

- 의도는 반전 메시지 전달이었지만
- 실제로 해당 지역 개발자들의 서버에서 데이터가 삭제됐다
- 주간 100만 건 이상 다운로드되던 패키지

## 4. 왜 암호화폐 관련 프로젝트가 특히 타겟인가

공격자 입장에서 생각해보자.

공급망 공격으로 얻을 수 있는 것:
- 개인정보 (이름, 이메일 등) → 팔 수 있지만 단가가 낮음
- 신용카드 정보 → 처리 과정이 복잡함
- 암호화폐 개인키 또는 지갑 정보 → **즉시, 익명으로, 되돌릴 수 없게 훔칠 수 있음**

암호화폐는 공격자에게 이상적인 타겟이다:
1. 국경이 없다 (어디서든 수령 가능)
2. 익명성이 있다 (추적이 어렵다)
3. 되돌릴 수 없다 (블록체인 트랜잭션은 취소 불가)
4. 즉시 현금화 가능하다

그래서 event-stream 사건에서도 타겟이 암호화폐 지갑이었고,
Web3 관련 패키지들은 공급망 공격의 주요 타겟이 된다.

## 5. 공격 방식의 종류

### 방식 1: 유지 관리자 계정 탈취

- npm 계정의 비밀번호가 약하거나 2FA가 없는 경우
- 피싱 등으로 계정 탈취
- 이후 악성 버전 배포

### 방식 2: 유지 관리자를 속여 권한 이전

- "내가 관리 도와주겠다"며 협조자로 들어온 뒤 악성 코드 추가
- event-stream 사례

### 방식 3: 타이포스쿼팅 (Typosquatting)

- 유명 패키지와 비슷한 이름의 패키지를 만들어 올림
- 예: `express` vs `expresss`, `lodash` vs `1odash` (L 대신 1)
- 개발자가 오타를 치면 악성 패키지가 설치됨

```bash
npm install lodash   # 정상
npm install 1odash   # 악성 패키지 (실제 존재했음)
```

### 방식 4: 의존성 혼란 공격 (Dependency Confusion)

- 내부 패키지 이름을 알아낸 뒤, 동일한 이름으로 npm 공개 레지스트리에 더 높은 버전을 올림
- npm이 공개 레지스트리의 버전을 우선시하는 경우 악성 패키지가 설치됨
- Microsoft, Apple, Tesla 등 대형 기업도 이 취약점에 노출됐음 (2021년 보안 연구자 데모)

## 6. 이 프로젝트에서 실천할 수 있는 것

### 원칙: 의존성은 부채다

패키지를 추가할 때마다 "이 패키지가 정말 필요한가?"를 먼저 물어야 한다.

의존성이 많아질수록:
- 공격 표면(attack surface)이 넓어진다
- 유지 관리해야 할 것이 늘어난다
- 버전 충돌이 발생할 가능성이 높아진다

편의를 위한 유틸 패키지 하나를 추가하는 게 10줄짜리 함수를 직접 짜는 것보다
항상 좋은 선택은 아니다.

### 실천 1: 패키지 최소화

이 프로젝트에 진짜 필요한 것:
- `express`: Node.js 웹 서버 프레임워크
- `ethers`: Ethereum RPC 통신
- `dotenv`: 환경변수 로드
- `cors`: Cross-Origin Resource Sharing 헤더 (React-Node 통신에 필요)
- (선택) `express-rate-limit`: Rate limiting

React 프론트엔드:
- `react`, `react-dom`: 기본
- `axios` 또는 기본 `fetch`: API 호출 (fetch는 브라우저 내장이라 설치 불필요)

이 외에 추가하고 싶은 패키지가 생기면 먼저 "직접 구현할 수 있는가?"를 물어볼 것.

### 실천 2: 패키지 신뢰도 확인

패키지를 추가하기 전 확인할 것:

```
1. npm 주간 다운로드 수: 많을수록 검증된 것
2. GitHub 레포지토리
   - 스타 수
   - 마지막 커밋 날짜 (오래됐으면 관리 안 되는 것)
   - 오픈 이슈 수
   - 유지 관리자 수 (1명이면 위험 신호)
3. 의존하는 패키지 수 (npm 페이지에서 확인 가능)
```

### 실천 3: package-lock.json 커밋

`package-lock.json`은 실제로 설치된 모든 패키지의 정확한 버전과 무결성 해시를 기록한다.

```json
{
  "name": "my-app",
  "lockfileVersion": 3,
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-...",   // ← 이 해시가 중요
      ...
    }
  }
}
```

`integrity` 해시는 패키지 파일의 SHA-512 해시다.
설치 시 이 해시를 검증하므로, 패키지가 변조됐으면 설치가 실패한다.

`package-lock.json`을 커밋하면:
- 다른 환경에서도 동일한 버전이 설치된다
- 패키지 버전이 슬금슬금 바뀌는 것을 방지한다
- CI/CD에서 재현 가능한 빌드가 가능하다

### 실천 4: npm audit

```bash
npm audit
```

알려진 취약점이 있는 패키지가 있으면 보고해준다.

```
found 2 vulnerabilities (1 moderate, 1 high)
  run `npm audit fix` to fix them, or `npm audit` for details
```

`npm audit fix`로 대부분 자동으로 수정 가능하다.
단, major 버전 업그레이드가 필요한 경우 breaking change가 있을 수 있으므로 주의.

## 7. .gitignore와 환경변수

공급망 공격과는 결이 다르지만, 같이 신경 써야 할 것:

**API 키를 코드에 하드코딩하지 말 것.**

나쁜 예:
```javascript
const provider = new ethers.JsonRpcProvider(
  "https://eth-sepolia.g.alchemy.com/v2/실제API키"  // 절대 안 됨
);
```

좋은 예:
```javascript
// .env 파일 (git에 커밋하지 않음)
ALCHEMY_URL=https://eth-sepolia.g.alchemy.com/v2/실제API키

// 코드
const provider = new ethers.JsonRpcProvider(process.env.ALCHEMY_URL);
```

`.env` 파일은 반드시 `.gitignore`에 포함시켜야 한다.
실수로 커밋됐다면, API 키를 즉시 재발급받아야 한다.
git history에 남은 키는 삭제해도 이미 노출된 것으로 봐야 한다.
